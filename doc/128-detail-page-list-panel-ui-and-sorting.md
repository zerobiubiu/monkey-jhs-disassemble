# 128 - 详情页清单面板 UI、排序与可靠交互优化

> **文档类型**：🔧开发指导  
> **文档状态**：✅已执行  
> **执行日期**：2026-07-19

## 1. 背景

视频详情页原清单区域只是把 JavDB 隐藏弹窗中的 checkbox 逐项克隆到一个空容器：

- 清单较多时缺少层级、搜索和统计，名称与数量不易扫读；
- 顺序完全服从服务端返回，无法按名称查找，用户要求默认名称降序并可切换；
- 加载、空数据、网络失败和不可操作状态没有明确反馈；
- modal 或 listContainer 被 JavDB 替换时，旧观察节点和按位置同步可能失效；
- `/users/simple_lists` 存在分页时，首屏不能代表用户的完整清单集合；
- 新建清单若仅依据模糊名称或页面瞬时 DOM 判断，可能把错误 ID 写入本地关联。

本次在不改变 JavDB 原生 checkbox 写入链的前提下，将详情页清单区升级为稳定、可检索、
可排序的渐进增强面板，并继续遵守 doc/126 的“服务端成功及权威复核后才能写本地”原则。

## 2. 方案

### 2.1 信息结构与排序

清单面板拆分为四个稳定区域：

1. 标题与“已加入 / 显示”汇总；
2. 默认“名称降序”的排序选择器，可切换“名称升序”；
3. 即时搜索框与独立清空按钮；
4. 只重绘条目的列表区，以及加载、空数据和错误状态区。

排序使用 `Intl.Collator('zh-CN', { numeric: true, sensitivity: 'base' })`，同名时以
`listId` 稳定决胜。搜索和排序只改变展示克隆的可见性与顺序，不改变 JavDB 原生 DOM，
也不影响服务端清单顺序。

### 2.2 完整分页与稳定同步

- 以 `/users/simple_lists?vid=...` 为权威清单源，跟随同源、同路径的 `rel=next` 聚合
  全部分页；设置 50 页上限、已访问 URL 防环、10 秒请求/响应体超时。
- 后续页条目先按 `listId` 去重，再补入隐藏 modal，所有操作仍由唯一的原生 checkbox
  发出；展示面板本身不会构造第二条服务端写入链。
- 观察稳定的 modal 外壳而不是一次性的 listContainer；容器被替换时递增 generation，
  旧分页结果不得覆盖新一代数据。
- 完整加载失败时保留已取得的数据并标记“部分清单”，初始化失败提供明确重试入口；
  source 消失、禁用或已过期时回滚展示 checkbox，并提示用户重试。
- 原生 checkbox 的 checked/disabled/busy 变化优先就地同步，完整重绘时恢复原焦点，
  避免键盘用户因服务端反馈丢失操作位置。

### 2.3 新建清单可靠性

- 完整分页基线未就绪时禁用“新建”，防止只看首屏而误判新 ID；
- 提交前记录完整 `beforeIds`，随后仅接纳不在基线中的候选 ID；
- 候选必须通过 `/users/simple_lists` 再次确认：规范化名称精确相等、当前影片
  `checked=true`、数量有效；多个候选或结果未知均不得写本地；
- `/users/lists` 回退路径同样使用规范化精确名称和唯一 ID，再执行上述权威复核；
- 成功后只提交已确认的单个 checkbox；失败或未知保留输入与焦点，不显示假成功；
- 新建表单、取消、Enter、输入法组合键、busy 和焦点恢复均有独立状态控制。

## 3. UI 与无障碍

- 延续 JavDB/Bulma 的克制工具型风格，不引入装饰性渐变或重阴影；
- 桌面端使用自适应双列、320px 可滚动条目区；移动端改为单列自然展开；
- 名称最多两行，数量为独立徽标，选中态和“同步中 / 已满 / 不可用”不只依赖颜色；
- 搜索、清空、重试及移动端条目具备清晰焦点和至少 44px 触控高度；
- `aria-live` 播报汇总和错误，`aria-busy` 表达加载/提交状态；
- 支持 `prefers-reduced-motion`，禁用非必要过渡。

以上取舍登记在根目录 `PRODUCT.md`，作为后续界面工作的产品与设计上下文。

## 6. 后续验证建议

1. 部署 v1.20.0 后，在真实账号上各执行一次加入/移出清单，确认提示、回滚及计数同步；
2. 使用清单超过一页的账号确认所有分页均可搜索和排序；
3. 创建一个唯一名称的新清单，再模拟断网失败，分别确认权威成功链和“未知不落本地”；
4. 在移动端实机验证长名称、软键盘搜索和滚动位置保持。
